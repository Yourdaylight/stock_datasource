## ADDED Requirements

### Requirement: Open API Scope Restriction (CRITICAL)
The Open API Gateway SHALL only expose interfaces generated by `ServiceGenerator` from plugin services (`src/stock_datasource/plugins/*/service.py`). The system SHALL obtain the list of exposable interfaces exclusively from `PluginManager`, and SHALL NOT scan global application routes, module routers, or API route files. All Module routes and standalone API routes are permanently excluded.

#### Scenario: Plugin route is accessible
- **WHEN** an external user requests `/api/open/v1/stock_daily/query` and `stock_daily` is a registered plugin
- **THEN** the request is processed (subject to API Key auth and access policy)

#### Scenario: Module route is blocked
- **WHEN** an external user attempts to access `/api/open/v1/market/kline` or any module-based path
- **THEN** the system returns HTTP 404 (the route does not exist in Open API namespace)

#### Scenario: AI/LLM endpoint is blocked
- **WHEN** an external user attempts to access `/api/open/v1/chat/send` or any AI-related path
- **THEN** the system returns HTTP 404 (the route does not exist in Open API namespace)

#### Scenario: System auth route is blocked
- **WHEN** an external user attempts to access `/api/open/v1/auth/login` or similar system paths
- **THEN** the system returns HTTP 404 (the route does not exist in Open API namespace)

#### Scenario: Unknown plugin is rejected
- **WHEN** an external user requests `/api/open/v1/nonexistent_plugin/query` where `nonexistent_plugin` is not registered in PluginManager
- **THEN** the system returns HTTP 404 with message indicating the plugin does not exist

### Requirement: Open API Key Authentication
The system SHALL authenticate external API requests using the existing MCP API Key (`sk-xxx`) mechanism. The system SHALL support two methods of providing the API Key: HTTP Header (`Authorization: Bearer sk-xxx`) and query parameter (`?api_key=sk-xxx`).

#### Scenario: Valid API Key via Header
- **WHEN** a request is sent to `/api/open/v1/{plugin}/{method}` with header `Authorization: Bearer sk-xxx` containing a valid, non-expired API Key
- **THEN** the request is authenticated and processed

#### Scenario: Valid API Key via Query Parameter
- **WHEN** a request is sent to `/api/open/v1/{plugin}/{method}?api_key=sk-xxx` with a valid API Key
- **THEN** the request is authenticated and processed

#### Scenario: Missing API Key
- **WHEN** a request is sent without an API Key
- **THEN** the system returns HTTP 401 with error message

#### Scenario: Invalid or Expired API Key
- **WHEN** a request is sent with an invalid or expired API Key
- **THEN** the system returns HTTP 401 with error message

### Requirement: API Access Policy Management
The system SHALL maintain an `api_access_policies` table that stores per-interface access control settings. Each policy SHALL include: enabled status, rate limits (per minute and per day), and maximum records per response. Only administrators SHALL be able to create, update, or delete policies.

#### Scenario: Admin enables an interface
- **WHEN** an admin sets `is_enabled=1` for api_path `stock_daily/query`
- **THEN** external users with valid API Keys can query that interface via `/api/open/v1/stock_daily/query`

#### Scenario: Disabled interface rejected
- **WHEN** an external user requests an interface where `is_enabled=0`
- **THEN** the system returns HTTP 403 with message indicating the interface is not available

#### Scenario: Non-admin cannot manage policies
- **WHEN** a non-admin user attempts to create or modify a policy
- **THEN** the system returns HTTP 403

### Requirement: Rate Limiting
The system SHALL enforce per-API-Key rate limits based on the policy configuration. Rate limits SHALL operate on two tiers: per-minute and per-day. When a limit is exceeded, the system SHALL return HTTP 429.

#### Scenario: Rate limit exceeded
- **WHEN** an API Key exceeds the `rate_limit_per_min` for a given interface
- **THEN** the system returns HTTP 429 with `Retry-After` header

#### Scenario: Daily limit exceeded
- **WHEN** an API Key exceeds the `rate_limit_per_day` for a given interface
- **THEN** the system returns HTTP 429 with message indicating daily limit reached

### Requirement: Response Record Limiting
The system SHALL enforce `max_records` from the access policy. When a query returns more records than `max_records`, the system SHALL truncate the result and include a `truncated: true` flag and `total_available` count in the response.

#### Scenario: Response exceeds max_records
- **WHEN** a query returns 8000 records but policy `max_records` is 5000
- **THEN** the response contains 5000 records with `truncated: true` and `total_available: 8000`

### Requirement: API Usage Logging
The system SHALL log every Open API call to the `api_usage_log` table. Each log entry SHALL include: api_path, api_type, user_id, api_key_id, record_count, response_time_ms, status_code, and client_ip. Logging SHALL be asynchronous (fire-and-forget) to avoid impacting response latency.

#### Scenario: Successful call logged
- **WHEN** an authenticated API call completes successfully
- **THEN** a log entry is written with status_code=200 and accurate response_time_ms

#### Scenario: Failed call logged
- **WHEN** an authenticated API call fails (e.g., invalid parameters)
- **THEN** a log entry is written with the error status_code and error_message

### Requirement: Open API Interface Discovery
The system SHALL provide a `GET /api/open/docs` endpoint that returns a list of all enabled interfaces with their descriptions, parameter schemas, and example requests. This endpoint SHALL NOT require authentication.

#### Scenario: Docs endpoint returns enabled interfaces
- **WHEN** a GET request is made to `/api/open/docs`
- **THEN** the response contains a list of all `is_enabled=1` interfaces with name, description, parameters, and example curl command

### Requirement: Admin API Access Management Panel
The system SHALL provide a frontend admin page at `/admin/api-access` that allows administrators to: view all discoverable interfaces, toggle enable/disable per interface, configure rate limits and max_records, and view usage statistics.

#### Scenario: Admin views interface list
- **WHEN** an admin navigates to `/admin/api-access`
- **THEN** all plugin interfaces are listed with their current enable/disable status

#### Scenario: Admin toggles interface
- **WHEN** an admin toggles an interface from disabled to enabled
- **THEN** the `api_access_policies` table is updated and the interface becomes available to external users

### Requirement: Internal Route Authentication
All HTTP routes that modify state (write operations) SHALL require user authentication via `get_current_user`. Destructive operations (cache flush, workflow delete) SHALL require `require_admin`. Plugin-generated routes from `ServiceGenerator` SHALL default to requiring authentication.

#### Scenario: Cache flush requires admin
- **WHEN** a non-admin user calls `POST /api/cache/flush`
- **THEN** the system returns HTTP 403

#### Scenario: Plugin route requires login
- **WHEN** an unauthenticated user calls any `POST /api/{plugin}/{method}` endpoint
- **THEN** the system returns HTTP 401
